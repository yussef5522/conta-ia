## 2. Decisões de arquitetura — preciso aprovação

### 2.1 React Query — usar ou não?

Spec da sprint pede "optimistic updates" no edit inline. Há 2 caminhos:

| Opção | Trade-off |
|---|---|
| **A (Recomendado)** Manual optimistic (useState set imediato + revert em erro) | Menos refactor — `page.tsx` continua com useState/useEffect. Padrão atual do projeto. |
| B Migrar pra `@tanstack/react-query` (já instalado) | Mais robusto a longo prazo. MAS refator grande no fetch pattern do projeto. Risco regressão alta. |

**Minha recomendação:** A. Pago "tecnicamente menos elegante" mas com risco MUITO menor de quebrar `/contas-a-pagar` já validado. Migração pra React Query é uma sprint separada que vale apena no futuro.

### 2.2 Faseamento interno

5 features grandes em 1 PR = alto risco. Proposta de faseamento em 5 commits validáveis (todos na mesma branch, deploy só no fim):

| Commit | Foco | Tempo |
|---|---|---|
| **c1** | Backend (migration + 5 CRUD endpoints + PATCH inline) | 1h |
| **c2** | Density toggle + Column hide/reorder (UI mais simples) | 1.5h |
| **c3** | Edit Inline (4 cell editors: text/number/date/combobox) | 2h |
| **c4** | Aging Dashboard | 1h |
| **c5** | Saved Views CRUD (tabs evolução + modais) | 1.5h |

**Alternativa:** 1 PR monolito (mesmo escopo, sem checkpoints). Seu chamado.

### 2.3 Endpoint Edit Inline

Spec sugere `PATCH /api/empresas/[id]/contas-pagar/[transactionId]/inline` com body `{ field, value }`.

**Concordo** — endpoint dedicado pra:
- Audit log granular (`User X editou campo "categoria" da tx Y`)
- Validação Zod estrita por campo (cada um tem regras diferentes)
- Reuso em /contas-receber e /movimentacoes (próximas sprints)

### 2.4 Optimistic update — escopo da reversão

Quando edit inline falha, revertemos APENAS a linha afetada (não a tabela toda). Implementação: state local da linha em edit; em erro, restaura state anterior; toast de erro.

### 2.5 Cell editor de Categoria — Combobox custom OU reuso

Não temos componente `<Combobox>` nem `<Command>` (cmdk fora do escopo). Opções:

| Opção | Trade-off |
|---|---|
| **A (Recomendado)** Input + dropdown filtrado custom | Sem dep nova. Funciona, mas UX menos polida. Suficiente pra sprint. |
| B Instalar `cmdk` (vai entrar em 5.0.3.0d pro Cmd+K também) | Adianta dep. Mas você disse "não fazer Cmd+K agora". |
| C Usar `<Select>` (shadcn/Radix) sem busca | Funciona pra <50 categorias, mas Cacula tem 245 — busca obrigatória |

**Recomendação:** A. Componente `<CategoryComboboxCell>` simples — Input com debounce + lista filtrada + "+ Criar 'X'" se nada bate.

### 2.6 Mobile responsive

Conforme spec: **NÃO** fazer mobile deep nesta sprint (vai pra 5.0.3.0d). Apenas garantir que NÃO QUEBRA em mobile:
- Density toggle: força "compact" em viewport <768px
- Column reorder: drag DESABILITADO em mobile

- Tabs custom: continuam scroll horizontal (já fizemos em 5.0.3.0b)

## 3. Detalhamento por commit

### Commit c1 — Backend (1h)

**Migration `20260607000000_sprint_5_0_3_0c_saved_view`:**
```sql
CREATE TABLE "saved_views" (
  "id" TEXT NOT NULL,
  "userId" TEXT NOT NULL,
  "empresaId" TEXT,
  "scope" TEXT NOT NULL DEFAULT 'payable',
  "name" VARCHAR(50) NOT NULL,
  "icon" VARCHAR(10),
  "filters" JSONB NOT NULL,
  "sortBy" TEXT,
  "sortDir" TEXT,
  "columnOrder" TEXT[] DEFAULT '{}',
  "columnHidden" TEXT[] DEFAULT '{}',
  "density" TEXT NOT NULL DEFAULT 'normal',
  "pinnedOrder" INTEGER NOT NULL DEFAULT 0,
  "createdAt" TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  "updatedAt" TIMESTAMP(3) NOT NULL,
  CONSTRAINT "saved_views_pkey" PRIMARY KEY ("id")
);
CREATE INDEX "saved_views_userId_empresaId_idx" ON "saved_views"("userId", "empresaId");
CREATE UNIQUE INDEX "saved_views_userId_empresaId_name_key" ON "saved_views"("userId", "empresaId", "name");
ALTER TABLE "saved_views" ADD CONSTRAINT "saved_views_userId_fkey"
  FOREIGN KEY ("userId") REFERENCES "users"("id") ON DELETE CASCADE;
```

Adicionar relação `User.savedViews` (sem alterar User direto, só Prisma model).

**Endpoints:**
- `GET /api/saved-views?empresaId=X&scope=payable` — lista do usuário
- `POST /api/saved-views` — criar (Zod validation, `name` único por userId+empresaId)
- `PATCH /api/saved-views/[id]` — editar (renomear/icon/filters/columns/density)
- `DELETE /api/saved-views/[id]` — excluir
- `POST /api/saved-views/[id]/duplicate` — duplicar com `name + " (cópia)"`
- `PATCH /api/saved-views/reorder` — batch `{ ids: [] }` → atualiza `pinnedOrder`
- `PATCH /api/empresas/[id]/contas-pagar/[transactionId]/inline` — edit cell

Multi-tenant: usuário só vê/edita SUAS próprias views (não as de outros usuários da mesma empresa). Endpoint inline edit reusa lógica de `/api/contas-a-pagar/[id]` PATCH com schema mais estrito.

**Tests c1:** ~15 — Zod schemas + audit log + multi-tenant guards.

## 5. Riscos identificados + mitigação

| Risco | Mitigação |
|---|---|
| @tanstack/react-table + @dnd-kit column reorder = combinação não trivial | Seguir example oficial de @dnd-kit pra tanstack v8. Teste manual em c2. |
| Optimistic update revert pode dessincronizar UI | `JSON.stringify(prev)` snapshot antes; `JSON.parse` no revert. Toast claro. |
| Edit inline + bulk selection no mesmo state pode conflitar | `editing.rowId` exclusive — entrar em edit limpa selection da linha. |
| Combobox de categorias pode lagar com 245+ categorias (Cacula) | Debounce 200ms + slice top 20 resultados |
| Migration `SavedView` em prod sob load — risco baixo | Migration aditiva, tabela vazia, deploy off-peak |
| Tests de localStorage exigem mock | vi.spyOn(window.localStorage, ...) — padrão Vitest |
| Mobile mod | Apenas garantir não-quebra; refactor mobile real em 5.0.3.0d |
