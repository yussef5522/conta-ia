## 2. Schema atual `ai_insights_log` (prod)

```
Column         | Type      | Nullable
---------------+-----------+----------
id             | text      | NOT NULL
companyId      | text      | NOT NULL
userId         | text      | NULL
feature        | text      | NOT NULL
model          | text      | NOT NULL
currentPeriod  | text      | NULL    ← YYYY-MM hoje
basePeriod     | text      | NULL    ← YYYY-MM hoje
inputTokens    | integer   | NOT NULL DEFAULT 0
outputTokens   | integer   | NOT NULL DEFAULT 0
costCents      | integer   | NOT NULL DEFAULT 0
elapsedMs      | integer   | NOT NULL DEFAULT 0
responseJson   | text      | NULL
errorMessage   | text      | NULL
createdAt      | timestamp | NOT NULL
```

**Decisão sobre os 5 campos novos da spec:**

A spec sugeriu adicionar `startDate`, `endDate`, `compareStartDate`, `compareEndDate`, `mode` em colunas separadas. Mas analisando:

- **Os campos `currentPeriod`/`basePeriod` já existem** como TEXT (YYYY-MM).
- **Migrar pra DATE seria refactor invasivo** com risco de quebrar logs antigos.
- **Solução proposta:** REPURPOSE os 2 campos existentes pra ISO YYYY-MM-DD (compatível por superset — `2026-05` parseia como `2026-05-01`), adicionar SÓ 3 campos novos:
  - `currentEndPeriod` TEXT NULL (YYYY-MM-DD — end date)
  - `baseEndPeriod` TEXT NULL (YYYY-MM-DD — end date do compare period)
  - `mode` TEXT NULL ('comparative' | 'evolution' | 'single')

Ou seja: `currentPeriod` vira "start", `basePeriod` vira "compare start", e adicionamos as 2 ends + mode.

**Por que isso é melhor que 5 novos:**
- Zero breaking change em logs antigos (`2026-05` parseia normal)
- Migration menor
- Cache lookup index existente continua funcional
- Repurpose semântico claro (start sempre presente, end opcional)

**Alternativa rejeitada:** adicionar 5 campos novos + ignorar os 2 antigos. Vira lixo no schema, requer dual-read na lib de cache.

## 3. Decisão sobre `date-fns`

A spec usa `date-fns` em vários lugares (`subMonths`, `startOfQuarter`, etc.).

**Status:** `date-fns` NÃO está em `package.json`.

**Decisão:** **Não instalar.** Já há helpers em `lib/relatorios/preview-queries.ts` (`startOfMonthUTC`, `endOfMonthUTC`, `addMonthsUTC`). Vou estender com `startOfQuarterUTC`, `endOfQuarterUTC`, `startOfYearUTC`, `endOfYearUTC`, `subYearsUTC`, `differenceInMonths`. Tudo nativo Date.

**Por que:** consistência do projeto (sem libs de data instaladas, helpers manuais multi-tenant safe). Adicionar `date-fns` (~5MB) só pra 7 presets é overkill.

## 4. Estratégia de prompts pro modo automático

A spec propõe **3 prompts separados** (SYSTEM_PROMPTS object com 3 keys).

**Alternativa que vou propor:** **UM system prompt único** que recebe o modo como contexto e adapta a saída. Mais simples + mantém shape `InsightOutput` consistente.

```typescript
SYSTEM_PROMPT = `Você é o IA Contador do CAIXAOS...

REGRAS DE TOM (PT-BR informal)...

REGRAS POR MODO (você recebe o modo nos dados):
- comparative: foque em DIFERENÇAS entre 2 períodos específicos
- evolution: foque em TENDÊNCIAS no período (3-12 meses)
- single: foque em COMPOSIÇÃO e SAÚDE financeira atual (1-2 meses)

Schema de saída idêntico em todos os modos.
`
```

E o user prompt traz `<modo_analise>{mode}</modo_analise>` no XML.

**Por quê:** se um dia adicionar `seasonal` ou `forecast`, é só adicionar mais uma regra no mesmo prompt em vez de criar 4º arquivo.

**MAS:** se você preferir 3 prompts separados (mais explícito por modo, mais fácil de iterar individualmente), eu sigo. Pergunto isso na seção 7.

## 5. Estratégia de coleta de dados por modo

`collectInsightData` precisa adaptar conforme modo:

- **comparative**: 2 períodos × DRE + variâncias + top categorias (igual hoje)
- **evolution**: N meses × DRE mensal + categorias emergentes/desaparecidas/recorrentes + topCategoriesTotalPeriod (com `monthsPresent`)
- **single**: 1 período × DRE + top categorias (sem comparação histórica)

A função precisa retornar shapes ligeiramente diferentes. Vou criar **3 funções específicas** + uma top-level que despacha pelo modo:

```typescript
collectInsightDataForMode(params, mode):
  switch(mode) {
    case 'comparative': return collectComparative(...)
    case 'evolution': return collectEvolution(...)
    case 'single': return collectSingle(...)
  }
```

E o tipo `InsightInputData` vira union discriminada:

```typescript
type InsightInputData =
  | { mode: 'comparative'; ...campos current + base + variances }
  | { mode: 'evolution'; ...campos months[] + emerging + disappeared }
  | { mode: 'single'; ...campos current + topCategories }
```

## 8. Riscos identificados

| Risco | Mitigação |
|---|---|
| Logs antigos têm `currentPeriod` em YYYY-MM (não YYYY-MM-DD) | Repurpose compatível: parser aceita ambos, `2026-05` vira `2026-05-01` |
| Migration trava produção | Migration só ADD COLUMN nullable — zero downtime |
| Cache key novo invalida cache antigo | OK — análises antigas ficam em log mas não retornam pra requests novos. Cache TTL 1h naturalmente expira |
| Prompts diferentes geram qualidade inconsistente | Checkpoint após Fase 2 — curl test 3 modos antes da UI |
| Período > 12 meses não bloqueado | Dupla validação (front + back) com Zod |
| `AIInsightsCard` renderizado em 2 lugares quebra | Removo do `/relatorios/page.tsx` e movo lógica pra `/analise-ia/analise-ia-client.tsx` — instância única |
