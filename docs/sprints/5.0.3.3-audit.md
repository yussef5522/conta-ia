## 🎯 TL;DR

3 causas raízes encontradas — explicam EXATAMENTE o que Yussef observou:

1. **Race condition:** `setCurrentEmpresa` dispara `router.refresh()` ANTES do fetch async `/api/empresas/atual` setar o cookie. Server lê cookie velho.
2. **Dashboard ignora cookie:** `/dashboard/page.tsx` lê só `searchParams.empresa` ou cai pra `empresas[0]`. Nunca consulta o cookie `current_empresa_id` que o switcher seta.
3. **Client pages não sincronizam URL state:** `/contas-a-pagar` faz `useState(searchParams.get('empresaId'))` apenas no INIT. Quando URL muda via `router.replace`, o state local fica stale (sem useEffect que reage).

**Importante:** investigação revelou que **React Query NÃO é usado no projeto**
(grep retornou zero ocorrências em `app/`, `components/`, `lib/`). A estratégia
A do spec ("invalidateQueries") não se aplica. O caminho real é cookie +
searchParams + useEffect de sincronização.

## Causa raiz #1 — Race condition no setCurrentEmpresa

**`lib/contexts/empresa-context.tsx:122-136`** (já alterada na sprint 5.0.3.2):

```typescript
const setCurrentEmpresa = useCallback((id: string) => {
  setCurrentEmpresaIdState(id)
  // ... localStorage ...
  void fetch('/api/empresas/atual', { method: 'POST', body: {...} })  // ASYNC fire-and-forget
  // ... 50ms depois ...
  router.refresh()  // server re-renderiza
})
```

`void fetch(...)` é **fire-and-forget**. Quando `router.refresh()` roda
imediatamente em sequência, o servidor pode ler o cookie ANTIGO porque
o POST ainda está em voo. Race condition explica:
- Dashboard às vezes funciona (depende da latência de rede)
- Resultado inconsistente entre usuários e tentativas

**Correção:** awaitar o fetch ANTES de navigate/refresh:

```typescript
const setCurrentEmpresa = useCallback(async (id: string) => {
  setCurrentEmpresaIdState(id)
  // localStorage sync (já existe)
  // Aguarda o cookie SETAR no server (não fire-and-forget mais)
  try {
    await fetch('/api/empresas/atual', { method: 'POST', body: {...} })
  } catch {
    // logging silencioso
  }
  // SÓ ENTÃO navega/refresh — cookie já está atualizado
  if (pathMatch) {
    router.push(...)
  } else if (hasEmpresaIdOrEmpresa) {
    router.replace(...)
  } else {
    router.refresh()
  }
}, [pathname, searchParams, router])
```

WorkspaceSwitcher chama `selectEmpresa(id)` que chama `setCurrentEmpresa(id)`.
Ambas assinaturas viram `async` — TypeScript pega callsites no build.

## Causa raiz #2 — Dashboard ignora cookie do switcher

**`app/(dashboard)/dashboard/page.tsx:103-106`** (Server Component):

```typescript
const empresaAtual =
  (empresaQueryId && empresas.find((e) => e.id === empresaQueryId)) || empresas[0]
```

Lê empresa de:
1. `searchParams.empresa` (query param `?empresa=`)
2. `empresas[0]` (primeira por createdAt ASC)

**NÃO LÊ o cookie httpOnly `current_empresa_id`** que o switcher seta via
`/api/empresas/atual`! Quando user navega de `/contas-a-pagar?empresaId=cacula`
pra `/dashboard` sem `?empresa=` na URL, dashboard pega `empresas[0]` sempre
(que é a primeira por ordem — NÃO a escolhida pelo user no switcher).

**Helper já existe:** `lib/auth/current-empresa-cookie.ts` exporta
`getCurrentEmpresaIdFromCookie()` — basta usar.

**Correção** (resolução em 3 níveis de prioridade):

```typescript
import { getCurrentEmpresaIdFromCookie } from '@/lib/auth/current-empresa-cookie'

const cookieEmpresaId = await getCurrentEmpresaIdFromCookie()

const empresaAtual =
  (empresaQueryId && empresas.find((e) => e.id === empresaQueryId)) ??
  (cookieEmpresaId && empresas.find((e) => e.id === cookieEmpresaId)) ??
  empresas[0]
```

Combinado com fix #1 (await fetch), quando user troca empresa no switcher:
1. Cookie atualiza no server (await garante)
2. `router.refresh()` dispara
3. Dashboard re-renderiza
4. Lê cookie novo → empresa correta

## Causa raiz #3 — Client pages não sincronizam URL state

**`/contas-a-pagar/page.tsx:154-156`** (Client Component):

```typescript
const [empresaId, setEmpresaId] = useState<string>(
  searchParams.get('empresaId') ?? '',
)
```

`useState` com initializer só roda no INIT. Quando o switcher faz
`router.replace(?empresaId=X)` em outra empresa, `searchParams` no Next
re-renderiza o componente, mas o `empresaId` state **não** atualiza
automaticamente — `useState` ignora o initializer em re-renders.

Sintoma observado: "precisa sair e voltar pra ver dados novos" — fechando
e abrindo a página, o initializer roda DE NOVO com searchParams atual.

**Correção:** useEffect que sincroniza state com searchParams quando muda:

```typescript
useEffect(() => {
  const urlEmpresaId = searchParams.get('empresaId') ?? ''
  if (urlEmpresaId && urlEmpresaId !== empresaId) {
    setEmpresaId(urlEmpresaId)
    // Resetar filtros que dependem da empresa (selection, paginação)
    setSelection({})
    setPage(1)
  }
}, [searchParams])
```

Aplicar em qualquer client page que lê empresaId via searchParams (vou
auditar quais).
