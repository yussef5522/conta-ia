## 1. Decisões de arquitetura que preciso aprovar com você

### 1.1 Rota — global vs per-empresa

A spec menciona `/empresas/[id]/contas-pagar` mas o layout mostra header sem `empresa` na URL. Hoje existem:
- `/contas-a-pagar` (global, com dropdown de empresa) ← atual, tem card PAGAS
- `/empresas/[id]/contas-pagar/` ← só importer/sub-rotas, NÃO tem lista

| Opção | Trade-off |
|---|---|
| **A (Recomendado)** Refazer `/contas-a-pagar` (mantém dropdown empresa) | Menor disruption, padrão atual, multi-empresa em 1 tela |
| B Criar `/empresas/[id]/contas-pagar/page.tsx` nova; global fica como agregado multi-empresa | Per-empresa fica deep-link friendly mas duplica UI |
| C Trocar global por per-empresa; dropdown vira navegação | Maior disruption, atalho do header precisa mudar |

**Minha recomendação:** **A**. Reescrever a rota global com URL state forte (`?empresaId=&view=&period=...`). Per-empresa pode ser feita depois se valer a pena.

### 1.4 Migration nova

```prisma
model SavedView {
  id           String   @id @default(cuid())
  userId       String
  companyId    String?  // null = view global do usuário (não escopada a empresa)
  scope        String   @default("payable") // payable/receivable/transactions/etc — pra reuso futuro
  name         String
  icon         String?
  filters      Json
  sortBy       String?
  sortDir      String?  // asc/desc
  columnOrder  String?  // JSON array
  columnHidden String?  // JSON array
  density      String   @default("normal") // compact/normal/comfortable
  pinnedOrder  Int      @default(0)
  isSystem     Boolean  @default(false) // 4 views default imutáveis
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
  user         User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  company      Company? @relation(fields: [companyId], references: [id], onDelete: Cascade)
  @@index([userId, scope, pinnedOrder])
  @@map("saved_views")
}
```

Sem índice em Transaction (`@@index([lifecycle, dueDate])` e `@@index([lifecycle, status])` já existem).

## 4. Riscos identificados

| Risco | Mitigação |
|---|---|
| `npm install` com `--legacy-peer-deps` (eslint conflict pré-existente) — pode quebrar build em prod | Test build local antes do deploy |
| @tanstack/react-table v8 com Next 16 + RSC — alguns components precisam 'use client' | Wrapper `<DataTable>` declara 'use client' upfront |
| Virtualization quebra accessibility (rows fora do DOM) | Usar `@tanstack/react-virtual` que tem opções aria |
| Command palette aberto causa scroll lock — afeta `/contas-pagar` outras interações | Test manual + Radix Dialog tem focus trap correto |
| Saved views ID custom user pode colidir com view system | Use `isSystem` flag + 4 views system não tem ID DB |
| Migration `SavedView` em prod sob load — risco baixo (tabela vazia) | Migration aditiva, deploy off-peak |
| Refactor grande → regressão em features OFX/Excel | Tests E2E cobrindo `/contas-pagar` desde 5.0.3.0a |
| Mobile responsive deixa pra última fase → vai apertar | Considerar fazer mobile junto com cada sub-sprint |
