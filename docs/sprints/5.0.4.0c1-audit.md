## 4. Estratégia de algoritmo de variâncias

### Threshold-based (proposta)

Conforme spec da sprint — algoritmo simples, transparente, defensável. Z-score fica pra Sprint 5.0.4.0c2 quando profit_sao_borja tiver 6+ meses.

**9 levels** conforme spec:
- `NEW` (categoria não existia no período base)
- `CRITICAL_UP` (+50% ou mais)
- `HIGH_UP` (+25 a +50%)
- `MODERATE_UP` (+15 a +25%)
- `STABLE` (±15%, **filtrado out** — não vira variância)
- `MODERATE_DOWN` (-15 a -25%)
- `HIGH_DOWN` (-25 a -50%)
- `CRITICAL_DOWN` (-50% ou pior)
- `DISAPPEARED` (existia, sumiu)

### Materiality filter

Default R$ 500 (env `VARIANCE_MIN_ABSOLUTE_VALUE`).

**Por que R$ 500:** ruído de café/material escritório que oscila 100% em valores baixos polui análise. Yussef pode ajustar depois.

### Edge cases tratados nos testes
- categoria nova com valor abaixo materiality (não alerta)
- categoria desaparecida (alerta)
- variação na fronteira exata (15%, 25%, 50%)
- previous = 0 (NEW, sem divisão por zero)
- current = 0 (DISAPPEARED)
- ambos zero (filtrado)
- valor negativo (sanidade, não devia acontecer mas defesa)

## 5. Estratégia Claude API

### Modelo
**`claude-sonnet-4-6`** conforme spec do Yussef.

### Pricing observado (Anthropic docs Abr/2026)
- Input: $3.00 / 1M tokens
- Output: $15.00 / 1M tokens

### Estimativa de custo por insight
- Prompt sistema: ~1.500 tokens
- Prompt user com variâncias + top categorias: ~2.000 tokens
- Output JSON: ~800 tokens
- **~$0.015/chamada (~R$ 0,08)** — controlável

Yussef escolheu **on-demand** (botão "Gerar análise da IA"). Cache 1h = no máximo 1 chamada/hora/empresa.

### Cache strategy
- Chave: `ai-insights:${empresaId}:${currentPeriod}:${basePeriod}`
- TTL: 1h
- Storage: **DB** (não Redis — projeto não tem Redis). Tabela `AICostLog` armazena resposta gerada como JSON pra cache também (campo `responseJson`).
- Invalidação: botão "Gerar novamente" força bypass de cache.

### Cost tracking
- Nova migration: `AICostLog`
- Logs: empresaId, userId, feature, model, inputTokens, outputTokens, costUSD, elapsedMs, createdAt, **responseJson** (cache + auditoria)
- Custo NUNCA exposto ao usuário (interno só)

### Streaming
**Não nesta sub-sprint.** Spec lista como "nice to have, não bloqueia entrega". Posso adicionar na 5.0.4.0c2 se Yussef pedir.

### Prompt engineering
Seguir template literal da spec (PT-BR informal, JSON estruturado, XML tags pra dados).

### Quality safety
- Validação Zod do JSON retornado pela IA antes de salvar.
- Se IA retornar estrutura inválida, devolve erro amigável (sem expor detalhes).

## 7. Riscos identificados

| Risco | Mitigação |
|---|---|
| API key vaza em logs | Sanitize errors no fetch client (já tem padrão) |
| Cache stale após user lança tx nova | Cache só por mês (currentPeriod) — period muda quando mês muda. Botão "Gerar novamente" bypassa pra usuário que acabou de editar |
| Cacula tem só 2 meses → variâncias travadas | Empty state amigável: "Precisamos de pelo menos 2 meses pra comparar. Importe contas dos meses anteriores" |
| IA inventa dados | Prompt veta explicitamente; validação Zod do JSON; XML tags estruturam input |
| Custo descontrolado | Cache 1h + botão on-demand + cost log permite auditoria; estimativa ~R$ 0,08/chamada |
| Sonnet 4.6 não disponível | Modelo confirmado nas docs Anthropic Jan/2026; fallback opcional pra Sonnet 4.5 se Anthropic deprecar |
| JSON malformado da IA | Regex extrai bloco `{}` (padrão atual do projeto); Zod valida shape; erro amigável se falhar |
