## Bug #1 — Conta R$ 100 fantasma (CRÍTICO)

### Evidência SQL

Transaction encontrada:
```
id:           cmpnjl1oe004uera2s242za39
amount:       100.00
description:  "yussef"
supplierId:   NULL
employeeId:   NULL
bankAccountId: NULL
categoryId:   cmpn1iaaf0024znevhuqdpc9p (categoria do Cacula Mix)
lifecycle:    PAYABLE
paymentDate:  NULL
dueDate:      2026-05-29
createdAt:    2026-05-27
```

Resolved company via JOIN: **cacula mix** (correto — categoryId aponta pra cacula).

### Confirmação: tx NÃO passa no whereBase de profit sao borja

SQL simulando o OR multi-tenant pra `empresaId = 'cmpnmgp9r0001px7oj7omv5y2'` (profit sao borja):
```
qty=0, total=NULL
```

Então o filtro multi-tenant **funciona** quando aplicado. O bug é no código.

### CAUSA RAIZ — `app/api/contas-a-pagar/route.ts:109-117`

```typescript
prisma.transaction.aggregate({
  where: {
    ...whereBase,    // contém { OR: [5 conditions multi-tenant] }
    status: 'PENDING',
    OR: [{ dueDate: { gte: now } }, { dueDate: null }],  // ← SOBRESCREVE OR multi-tenant!
  },
  _sum: { amount: true },
  _count: { _all: true },
})
```

**JavaScript object spread + override:** `...whereBase` espalha o `OR` multi-tenant, depois `OR: [{ dueDate: ... }]` na MESMA chave OR **apaga** o anterior. Resultado: aggregate roda em **TODAS as PAYABLE PENDING do banco** com dueDate>=now ou null → inclui a tx R$ 100 do Cacula no card da profit sao borja.

### Outros 3 KPIs (não afetados)

- **PAGAS** (linha 103): `{ ...whereBase, paymentDate: { not: null } }` — sem OR, NÃO sobrescreve → OK
- **VENCIDAS** (linha 119): `{ ...whereBase, status: 'PENDING', dueDate: { lt: now } }` — sem OR → OK
- **A VENCER 3d** (linha 125): `{ ...whereBase, status: 'PENDING', dueDate: { gte, lte } }` — sem OR → OK

Só "A pagar pendente" tem o bug.

### Correção proposta

Trocar `OR` na mesma chave por `AND` explícito, OU usar negação:

```typescript
// FIX (recomendado): AND explícito preserva OR multi-tenant
prisma.transaction.aggregate({
  where: {
    AND: [
      whereBase, // mantém OR multi-tenant intacto
      { status: 'PENDING' },
      { OR: [{ dueDate: { gte: now } }, { dueDate: null }] },
    ],
  },
  _sum: { amount: true },
  _count: { _all: true },
})
```

Tests novos: confirmar via SQL/unit que aggregate retorna 0 pra empresa SEM PAYABLE PENDING.

## Bug #2 — Cards ignoram filtro de data (CRÍTICO confiabilidade)

### Causa raiz — design intencional na Sprint 5.0.3.0a

`app/api/contas-a-pagar/route.ts:54-66`:
```typescript
const kpiBaseInput = {
  ...input,
  status: undefined,
  vencidasOnly: false,
  dataDe: undefined,   // ← LIMPA filtro de período pros KPIs
  dataAte: undefined,
  q: undefined,
}
const whereBase = buildPayableListWhere(kpiBaseInput, now)
```

Decisão original: "KPIs são snapshot atual da empresa". Yussef agora discorda — KPIs devem refletir o que o user filtrou.

### Correção proposta

Remover `dataDe: undefined` e `dataAte: undefined` do `kpiBaseInput`. Manter limpeza de `status/vencidasOnly/q` (essas SIM precisam ser limpas pra KPIs serem segmentados).

Resultado: KPIs respeitam dataDe/dataAte/dataField igual à tabela.

Mas atenção — **A VENCER 3d** sempre é "próximos 3 dias da data ATUAL", independente do filtro do user. Vou MANTER hard-code de "now → now+3d" nesse específico, OU passar a respeitar período? Spec do Yussef: "TODOS os 4 cards respeitam o filtro de período". Vou respeitar a spec literal.

**Refinamento:** se user filtrou 01/04→27/05, "A VENCER 3d" mostra tx com dueDate dentro de (max(now, 01/04) → min(now+3d, 27/05)). Pode dar 0 se 27/05 já passou. Aceito.
