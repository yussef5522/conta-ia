## 6. Decisões arquiteturais que precisam aprovação

### 6.1 Reuso vs Refactor do DRE

✅ **Recomendação: REUSAR (Opção A)** — risco zero, paridade automática, 30min vs 3h refactor.

### 6.2 Onde exatamente fica o DRE migrado?

Spec menciona `/empresas/[id]/relatorios/dre-gerencial`. Mas o DRE atual lê empresa
do COOKIE (rota global `/dre`). Opções:

| Opção | URL canônica | Trade-off |
|---|---|---|
| A | `/relatorios/dre-gerencial` (global, lê cookie) | Consistente com `/dre` atual. Mais simples. |
| B | `/empresas/[id]/relatorios/dre-gerencial` (per-empresa) | Deep-link friendly. Per-spec. |
| C | Ambos (per-empresa default + alias global) | Maior compat. Mais código. |

**Recomendação: B (per-empresa)** — alinha com spec + workspace switcher já navega pra `/empresas/[id]/*` quando user troca. `/dre` global vira redirect.

### 6.3 Comparativo 3 meses — fonte temporal

Pra calcular evolução: usar `paymentDate` (regime caixa) ou `competenceDate` (regime competência)?

**Recomendação:** **competenceDate** com fallback pra `date` (consistente com DRE existente em regime competência default). Filtro de "regime" na UI permite alternar.

### 6.4 Migração DRE — quando deletar arquivos antigos

**NÃO deletar arquivos do DRE existente** (Opção A é reuso, não move). Só:
- `/dre/page.tsx` → vira redirect (1 linha mantida)
- `/empresas/[id]/dre/page.tsx` → mantém redirect pra `/relatorios/dre-gerencial`
- Arquivos do DREClient + lib continuam ali (são reusados)
